# Learned Memory

Ajanlar bu dosyaya proje boyunca öğrendikleri non-obvious teknik bilgileri yazar.
Her oturumda otomatik okunur ve ajanlara enjekte edilir.

Format: `- [Kategori] Öğrenilen şey`

- [Backend] EmsCrmCustomer modelinde tarih alanı `CreatedAt` olmalı (NOT `CreatedOn`) — EMS API `createdAt` camelCase döner; `JsonNamingPolicy.CamelCase` ile C# `CreatedAt` ↔ JSON `createdAt` eşleşir, `CreatedOn` eşleşmez (field name farkı, case-insensitive eşleşmesi de yoktur).
- [Backend] GenericRepository her UpdateAsync/AddAsync çağrısında SaveChangesAsync yapar. Birden fazla entity'yi tek transaction'da güncellemek için Context.Database.BeginTransactionAsync() + ExecuteSqlRawAsync kullan, repository metodlarını zincirleme.
- [Backend] ContactHistory, CustomerTask ve Opportunity'de denormalized ProjectId var. CustomerId taşıma işleminde ProjectId aynı kalır (aynı tenant), sadece CustomerId güncellenir.
- [Backend] ResultToResponse(Result) başarıda 200 OK döndürür. Endpoint 204 NoContent istiyorsa controller'da `if (result.IsSuccess) return NoContent();` şeklinde özel handle et.
- [Backend] TransferLead için Command+Handler önceden scaffold edilmişti ama ICustomerRepository.TransferLeadAsync interface metodu ve implementasyonu eksikti — sadece bunları tamamlamak yetti.
- [Frontend] useCustomers hook'u backend status filter'ını destekler ama "Lead olmayan tüm müşteriler" için tek bir status query param yok — birden fazla status için client-side filter (c.status !== 'Lead') kullan.
- [Backend] Projects entity'sine yeni alan eklerken hem `Project.cs`'e property hem de `Program.cs` startup SQL bloğuna `ALTER TABLE "Projects" ADD COLUMN IF NOT EXISTS ...` eklemek gerekiyor. EF migration dosyası yok, schema değişiklikleri tamamen idempotent raw SQL ile yönetiliyor.
- [Backend] Yeni bir route prefix'i gereken endpoint (örn. /api/v1/crm/companies) için `ApiControllerBase`'den kalıtan yeni bir controller oluşturmak yeterli — base controller `[Authorize]` ve MediatR'ı içeriyor.
- [Backend] Paraşüt entegrasyonu iki katmanda çalışır: `IParasutClient` (düşük seviye HTTP/OAuth) ve `IParasutService` (yüksek seviye façade). Handler'lar sadece `IParasutService` inject eder; token yönetimi ve connection lookup gizlidir. `ParasutAutoConnectService` startup'ta token'ları warm eder (IgnoreQueryFilters() kullanır).
- [Backend] `ParasutConnection` entity'si per-project OAuth credentials'ı plaintext saklar (ClientId, ClientSecret, Username, Password). `ParasutConnectionConfiguration`'da `HasIndex(c => c.ProjectId).IsUnique()` ile bir projede tek bağlantı garantilenir.
- [Backend] `ParasutAutoConnectService` artık `BackgroundService` (periodic 30dk) — sadece startup'ta değil, runtime'da da expired token'ları otomatik yeniler. `PeriodicTimer` kullanır, `MaxReconnectAttempts=10` sonrasında durur. `OperationCanceledException` graceful shutdown için yakalanmalı.
- [Backend] Paraşüt cari hareketleri endpoint'i (`contact_debit_credit_transactions`) hem borç (fatura/debit) hem alacak (tahsilat/credit) işlemlerini döner. `IParasutService.GetContactTransactionsAsync` ile erişilir. İki farklı API çağrısı (invoices + transactions) `Task.WhenAll` ile paralel yapılır, partial failure'da kısmi veri döner.
- [Backend] Invoice entity LinesJson'ı denormalized JSON string olarak saklar. Paraşüt'e transfer sırasında `JsonSerializer.Deserialize` ile parse edilip `SalesInvoiceDetailData` listesine dönüştürülür. PropertyNameCaseInsensitive gerekli.
- [Backend] CRM Invoice → Paraşüt transfer akışında `ParasutTokenHelper.EnsureValidTokenAsync` doğrudan `IParasutClient` + `IParasutConnectionRepository` ile çağrılır (IParasutService yerine), çünkü mevcut CreateParasutSalesInvoiceCommandHandler da aynı pattern'ı kullanıyor.
- [Backend] Paraşüt e-fatura mükellef sorgusu: `GET /v4/{company_id}/e_invoice_inboxes?filter[vkn]={vkn}` — boş liste dönerse müşteri e-fatura mükellefi değil. E-fatura bilgisi best-effort olarak çekilir; hata durumunda ana işlem (fatura transferi) fail olmaz.
- [Backend] Paraşüt sales invoice list endpoint'i (`GetSalesInvoicesAsync`) attributes'ta contact/relationship bilgisi dönmez. Customer eşleştirmeli import için `GetContactInvoicesAsync` (filter[contact_id]) kullanmak gerekir — her customer için ayrı API call.
- [Backend] Paraşüt GrossTotal/NetTotal gibi parasal alanlar JSON'da string olarak döner (örn. "1234.56"). `decimal.TryParse` ile `CultureInfo.InvariantCulture` kullanarak parse etmek gerekir.
- [Backend] InvoiceRepository sorgularında `OrderByDescending(IssueDate)` kullanılmalı — import edilen faturalar aynı CreatedAt'e sahip olacağından CreatedAt sıralaması anlamsız olur.
- [Backend] Paraşüt e-fatura resmileştirme (officialize): e-invoice mükellefi → `POST /v4/{company_id}/e_invoices`, değilse → `POST /v4/{company_id}/e_archives`. Her iki endpoint de JSON:API relationship ile `sales_invoices` ID'sine bağlanır. Response'da `ParasutEInvoiceAttributes` (uuid, status, scenario) döner.
- [Backend] CreateInvoice auto-transfer+officialize best-effort pattern: ana işlem (DB'ye kaydet) her zaman başarılı olur, Paraşüt transfer/officialize hataları loglanır ama Result'a yansımaz. GenericRepository her AddAsync'te SaveChanges yaptığı için UpdateAsync ayrı bir call olarak yapılır.
- [Backend] Paraşüt satış faturası kalem (SalesInvoiceDetailData) → ürün (ürün/hizmet) ilişkisi: `relationships.product.data = { type: "products", id: "..." }` ile bağlanır. Entity'de `TaxRate` 0-1 decimal (örn. 0.20) olarak saklanır; Paraşüt API `vat_rate` integer (örn. 20) bekler → `(int)(TaxRate * 100)` ile dönüştür.
- [Backend] `IParasutProductRepository` DI olarak `DependencyInjection.cs`'teki Paraşüt bloğuna `services.AddScoped<IParasutProductRepository, ParasutProductRepository>()` ile eklendi. Upsert pattern: `GetByNameAsync` ile var mı kontrol et → varsa Update, yoksa Add.
- [Backend] `[JsonIgnore(Condition = WhenWritingNull)]` ile null olan optional relationship'lar Paraşüt'e gönderilmez — bu önemli, aksi halde API null relationship kabul etmeyebilir.
- [Frontend] Paraşüt ürün listesi (live API) lazy yüklenmeli: sayfaya girildiğinde değil, kullanıcı "Yükle" butonuna tıkladığında `enabled` flag'i set edilerek fetch tetiklenmeli. Bu, bağlantısız projelerde gereksiz API çağrısını ve hata toast'larını önler.
- [Frontend] Radix Select dropdown içinde arama desteği yok; searchable combobox için native `<input>` + controlled `open` state + overlay div (click-outside kapama) pattern'ı daha sade ve daha az bundle boyutlu bir alternatiftir — `cmdk` veya `@radix-ui/react-popover` eklemeye gerek kalmaz.
- [Frontend] Paraşüt ürün seçildiğinde `unitPrice` ve `vatRate` alanları auto-fill edilebilir; ancak kullanıcı zaten bir değer girmişse üzerine yazılmamalı (sadece boşsa doldur).
- [Frontend] EMS müşteri tespiti: `legacyId` numeric ise (`/^\d/.test(legacyId)`) veya `SAASA-` prefix'i varsa EMS müşterisi. `PC-` prefix'i manuel oluşturulmuş, `SAASB-` ise RezervAl müşterisi — bunlar EMS değil.
- [Frontend] Conditional tab query hook'u `enabled` param ile lazy yüklenebilir: `useCustomerEmsUsers(customerId, activeTab === 'ems-users' && isEmsCustomer(...))` — tab'a tıklanmadan API çağrısı yapılmaz.
- [Backend] RezervAl Create/Update multipart/form-data gönderiminde `MultipartFormDataContent` her field için `new StringContent(value)` ekler; logo `ByteArrayContent` olarak `Content-Type: application/octet-stream` ile eklenir. `bool` değerleri `"true"/"false"` string olarak gönderilmeli.
- [Backend] RezervAl `push-to-rezerval` endpoint'i create vs. update kararını LegacyId'den alır: "REZV-{n}" → PUT /v1/Crm/Company/{n}, diğer → POST /v1/Crm/Company. Create sonrası dönen `CompanyId` LegacyId olarak "REZV-{id}" formatında kaydedilir.
- [Tests] SaasSyncJob'un eski `SyncSaasAAsync` metodu (GetCustomersAsync çağırırdı) artık RunAsync'te comment'e alınmış. Aktif EMS sync `SyncEmsCrmCustomersAsync` + `GetCrmCustomersPageAsync` kullanıyor. Buna bağlı eski testleri `GetCrmCustomersPageAsync` verify'a güncelle.
- [Tests] SyncRezervalCompaniesAsync integration testi için SaasSyncJob doğrudan test edilmeli (RunAsync → SyncRezervalCompaniesAsync chain). ScopeFactory mock'u hem ISyncLogRepository hem ApplicationDbContext döndürmeli — aynı SetupScopeFactory helper pattern'ı kullan.
- [Tests] PushCustomerToRezervalCommandHandler'da update senaryosunda `_customerRepository.UpdateAsync` çağrılmaz (LegacyId zaten REZV-prefix, değişmez). Yalnızca create senaryosunda yeni LegacyId kaydedilir → UpdateAsync çağrılır.
